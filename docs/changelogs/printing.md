## 865. ✅ docker-compose.worker.yml + print-agent/Dockerfile shipping artifact
**Why:** Implements item 7 of the plan's code order — the operator-facing deployment artifact for a physical location. Deliberately a separate lightweight `print-agent/Dockerfile` rather than reusing the main `Dockerfile`: `backend/print-agent.ts` only imports `http`/`https`/`os`/`path`/`fs`/`child_process`/`ws` (no DB, no chromium, no cups-client.ts — by design, see its file-header comment), so a runtime image built from the main Dockerfile would needlessly ship the whole app's dependency tree (postgres client, chromium, etc.) onto a possibly-unguarded remote machine. Instead `print-agent/Dockerfile` is a two-stage build: a builder stage with the full root `package.json`/`tsconfig.json` (needed for `@types/node` etc. during `tsc`, discarded after), and a runtime stage that installs only `print-agent/package.json`'s slim `ws` dependency alongside the compiled `print-agent.js`. Verified locally (no Docker daemon available in this sandbox): copied `backend/`+`models/`+`types/`+`tsconfig.json` into a scratch dir, ran `npm ci && tsc`, confirmed `dist/backend/print-agent.js` emits despite the pre-existing unrelated `remove-item.ts` type error (tsc doesn't block emit on it), and that the emitted file loads and exits cleanly when required env vars are absent. `docker-compose.worker.yml` itself validated via `docker compose config`. The compose file documents both CUPS options up front: the bundled `cups` service (now also exposing port 631, since this is the worker's own local-only instance, not the cloud app's shared one — no conflict with the deferred port-631-on-the-shared-container item below), or pointing `CUPS_HOST` at an existing reachable CUPS server (e.g. the Raspberry Pi setup in `docs/detailed/printer-server-setup.md`) by deleting the `cups` service block — confirmed feasible because `print-agent.ts` only ever does `lp -h $CUPS_HOST -d <queue> ...`, no local socket assumption. `.env.worker.example` added alongside, mirroring the existing `.env.example` convention.
**Deferred:** Registry image publishing (`ghcr.io/...`) per the original plan text was not set up — no CI/registry workflow exists in this repo yet, so the compose file builds locally (`docker compose -f docker-compose.worker.yml up -d --build`) instead; revisit once a release pipeline exists. The shared (cloud-side) CUPS container's port 631 + basic auth and retiring `backend/utils/sync-printer-queues.ts` remain deferred for the same reason as #863 (no worker actually deployed yet to replace the still-live `PrinterQueuesCard.tsx` admin CRUD). Not yet tested against real hardware/printer or an actual Docker daemon — typecheck/emit and `docker compose config` validation only.

## 863. ✅ Admin "Worker nodes" view: agent hello auto-upserts discovered queues, Site/LabelTypes editing, "Queues abrufen"
**Why:** Closes the remaining gap in the multi-instance plan's per-printer routing config: previously a discovered agent queue had no UI to set `Site`/`LabelTypes`, so `resolvePrinterQueue` could never find a match for it. `agentServer.ts`'s `hello` handler now calls a new `db.ts` `upsertDiscoveredQueue(instanceId, name)` for each reported queue — INSERT/upsert into `printer_queues`, defaulting `site` to an existing sibling queue's site (if the instance already has one configured) or to the `instanceId` itself otherwise, matching the plan's stated default rules; `site`/`label_types` are never clobbered on reconnect since the upsert only writes `instance_id`/`updated_at` on conflict. New `GET /api/admin/worker-agents` (connected instance ids) and `PUT /api/admin/printer-queues/:name/routing` (admin sets `site`/`labelTypes`, JSON-encoded TEXT matching the existing `Meta`-column convention) endpoints, plus a `POST /api/admin/worker-agents/gather` that broadcasts `{type:'scan_devices'}` to all connected agents over the existing `/agent` WebSocket — `print-agent.ts` now replies by resending its `hello` message, which re-triggers the upsert without a full reconnect. New `WorkerNodesCard.tsx` admin card lists discovered queues with live online/offline status (`isAgentConnected`), an editable `Site` text field, and a `LabelTypes` checkbox group, wired into `AdminPage.tsx` alongside the existing `PrinterQueuesCard`. Deliberately a separate card rather than extending `PrinterQueuesCard` — that card's CRUD is device_uri/ppd_model/lpadmin-push-based (config owned by the cloud app), while Worker nodes queues are config owned by the local CUPS instance and only ever flow upward (discovery), keeping the two models visually distinct per the plan's Component 2 note.
**Deferred:** CUPS container port 631 + basic auth, `docker-compose.worker.yml`, and retiring `backend/utils/sync-printer-queues.ts` are still open — explicitly *not* done this round because `sync-printer-queues.ts`'s `lpadmin` calls currently rely on the unauthenticated CUPS `/admin` access that basic-auth would break, and no worker/agent is deployed yet to replace the still-live, currently-functioning printer-queue admin CRUD (`PrinterQueuesCard.tsx`/`admin-printer-queues.ts`) that depends on it. Retiring it now would regress production printing before the replacement exists — deferred until workers actually deploy, per the plan's own transition sequence. "Gather queues" re-discovery only re-sends the agent's static `AGENT_QUEUES` env list (no live `lpstat -p` rescan on the agent side yet) — sufficient for now since `print-agent.ts` doesn't dynamically detect new printers either.

## 862. ✅ Gate legacy runPrintWorker to untargeted jobs only; add operator `site` (lib/user.ts) sent with print requests
**Why:** Closes two gaps left open from #861. First, a correctness gap: the cloud app's legacy `runPrintWorker()` loop in `server.ts` still called the unrestricted `claimNextLabelJob()`, so it could claim a `TargetQueue`-tagged job meant for a remote print-agent if it happened to have a same-named local CUPS queue, racing the agent for its own job. Fixed by switching to `claimNextLabelJobForAgent([])` — the existing `WHERE "TargetQueue" IS NULL OR "TargetQueue" = ANY($2)` clause naturally restricts an empty `ownedQueues` array to untargeted jobs only, since `= ANY('{}')` is always false. The now-fully-unused `claimNextLabelJob()` was removed from `db.ts`. Second, a dead-code gap: nothing in the browser ever sent a non-empty `site`, so the per-site `resolvePrinterQueue` routing built in #860 was unreachable. Fixed by adding `getSite()`/`setSite()` to `frontend/src/lib/user.ts`, reusing the existing `username` localStorage pattern (`SITE_STORAGE_KEY`, same "ask once, remember, editable any time" UX) rather than building a new UI surface, per the plan's explicit design decision. `Header.tsx`'s double-click handler now chains a second `dialog.prompt()` call for the site after the username prompt (the existing `dialogService.prompt()` API only supports one value per call). All three call sites of `requestPrintLabel` (`PrintLabelButton.tsx`, `ItemCreate.tsx`, `RelocateItemCard.tsx`) now pass `site: getSite().trim() || undefined`, keeping the field optional so unconfigured/single-site deployments are unaffected.
**Deferred:** Admin "Worker nodes" UI (per-queue `Site`/`LabelTypes` editing, "Gather queues" button), CUPS container port 631 + basic auth, `docker-compose.worker.yml`, retiring `backend/utils/sync-printer-queues.ts` — all still open per the plan.

## 861. ✅ print-agent.ts standalone process + AGENT_TOKEN HTTP claim-job/status API
**Why:** Continues the multi-instance/multi-location plan (`docs/PLANNING_multi_instance.md`, Component 2) by adding the piece that actually prints: `backend/print-agent.ts`, a standalone Node entrypoint that connects to the cloud app's `/agent` WebSocket for liveness + `job_available` wake-ups, claims a job over a new AGENT_TOKEN-authenticated HTTP API, prints it via a local `lp` invocation against `CUPS_HOST`, and reports status back — all with exponential-backoff reconnect and a 30s fallback poll as a backstop for a dropped `job_available` message. Deliberately deviates from the plan's literal wording that the agent "reuses `cups-client.ts` and `print.ts` directly": both of those modules call the DB-backed `getSetting()` internally (`runCupsCommand()` reads `printer.server` from `app_settings` on every CUPS call), which conflicts with the plan's own hard constraint that the agent must hold no DB credentials. Resolved by adding two new AGENT_TOKEN-gated HTTP endpoints in the cloud app instead — `backend/actions/agent-claim-job.ts` (atomically claims a job restricted to the agent's queues via the existing `claimNextLabelJobForAgent`, renders the item label HTML server-side where DB/render access already lives, and returns the rendered HTML) and `backend/actions/agent-job-status.ts` (`POST /api/agent/jobs/:id/status`) — so the agent process itself only ever needs `APP_URL`, `AGENT_TOKEN`, `INSTANCE_ID` and `CUPS_HOST`/`AGENT_QUEUES`, no `DATABASE_URL`. New `backend/utils/agent-auth.ts` mirrors the existing `requireAdminAuth` bearer-token pattern for these endpoints. 7 new tests in `test/agent-job-actions.test.ts` cover auth rejection, empty-queue claims, successful render/claim, and the item-not-found error path.
**Deferred:** Non-item label types (box/shelf/marketingsheet) on the agent claim path — only `item` labels are rendered server-side for now, matching the scope of the pre-existing `runPrintWorker()` loop this supersedes for remote agents. The legacy `runPrintWorker()` loop in `server.ts` is not yet retired/gated, so the cloud app could still locally claim a `TargetQueue`-tagged job meant for a remote agent if it happens to have a same-named local CUPS queue — needs reconciling once agents are actually deployed. Also still deferred from #860: admin "Worker nodes" UI, `frontend/src/lib/user.ts` `site` field, CUPS container port 631 + basic auth, `docker-compose.worker.yml`, retiring `sync-printer-queues.ts`.

## 860. ✅ Multi-location print routing foundation: TargetQueue schema, per-site queue resolution, /agent WebSocket control plane
**Why:** Implements the first steps of the approved multi-instance/multi-location deployment plan (`docs/PLANNING_multi_instance.md`): a single cloud app + DB, with lightweight local print agents at each physical site connected over a WebSocket control plane (no DB credentials on unguarded local machines). Added: `label_queue.TargetQueue` (routes a queued job to a specific printer queue) and `printer_queues.{instance_id,site,label_types}` (additive `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` columns in `initDb()`); `queueLabel(itemUUID, targetQueue?)` and `claimNextLabelJobForAgent(ownedQueues)` (a worker only claims jobs targeting its own queues, or untargeted jobs, for backward compat with un-configured single-site setups); `resolvePrinterQueue(labelType, site?)` now first checks `printer_queues` for a row matching the operator's `site` whose `label_types` (JSON-in-TEXT, matching the existing `Meta`-column convention rather than introducing a new array/jsonb column type) includes the requested label type and whose owning agent is currently connected, before falling back to the existing `app_settings`/env-var default queue (so single-site deployments are unaffected). New `backend/agentConnections.ts` holds the shared `connectedAgents` map so both the WebSocket server and `print.ts` can read agent liveness without a circular import. New `backend/agentServer.ts` adds a `/agent` WebSocket endpoint (using the new `ws` dependency), validated by a single shared `AGENT_TOKEN` at handshake — deliberately one token for all workers for now (documented tradeoff: simpler rollout, but a stolen token from one unguarded site could connect as any `instanceId`; revisit per-agent tokens if a site is known compromised or worker count grows). `print-unified.ts` now forwards an optional `site` field from the print request body through to `resolvePrinterQueue`, mirroring the existing `actor` field.

## 859. ✅ Atomic label-queue claim (FOR UPDATE SKIP LOCKED) — enables cross-instance printing
**Why:** `nextLabelJob()` issued an unlocked SELECT; with multiple instances each polling every 750 ms, two workers could claim the same row and print the same label twice. Replaced with `claimNextLabelJob()` — an atomic UPDATE…RETURNING with a subquery using `FOR UPDATE SKIP LOCKED`, which transitions the row to `Processing` in the same statement. Added `recoverStaleLabelJobs()` to reset Processing rows older than 5 minutes back to Queued (crash recovery). Added `ClaimedAt` TEXT column to `label_queue` (via `ALTER TABLE … ADD COLUMN IF NOT EXISTS`). Side-effect: all instances share one label queue through Postgres, so a print request from any instance is fulfilled by whichever worker is free — printing to that instance's local CUPS printer. Admin endpoint (`/api/admin/label-queue`) now also reports `processing` count.
**Deferred:** Explicit printer routing (a `TargetPrinter` column to direct a job to a specific named printer) — implicit routing via whichever instance claims the job is sufficient for now. Unit test for double-claim prevention requires a live Postgres DB.

## 853. ✅ Fix CUPS fd limit (ulimits) and www-data lpinfo Forbidden error
   - **Why:** (1) CUPS crashes on hosts with high/unlimited fd limits — added `ulimits.nofile: 65000/65000` to the cups service in both compose files; CUPS internal calculations assume lower limits and produce an invalid value when the limit is too high, causing EFAULT. (2) www-data (uid 33) in the mediator container gets "Forbidden" from `lpinfo` via the Unix socket — `AuthType None` in cupsd.conf disables credential prompts but CUPS still enforces lpadmin group membership at the socket level; fixed by adding `www-data` to `lpadmin` in `cups/Dockerfile`. USB device passthrough was already handled by docker-compose.usb.yml.
   - **Deferred:** Nothing.

## 823. ✅ Fix lpadmin Forbidden + printer_not_ready: cupsd Policy block + empty ppd_model guard
   - **Why:** CUPS has two auth layers — `<Location>` (network/IP) and `<Policy>` (operation/user). `AuthType None` on `/admin` fixed the first; the default `<Policy default>` still required `@SYSTEM` for printer management, causing 403 Forbidden from www-data even via socket. ## 822. ✅ Fix lpinfo "Bad Request" (CUPS 2.4): file-based device/PPD discovery
   - **Why:** CUPS 2.4 removed `CUPS-Get-Devices` and `CUPS-Get-PPDs` IPP operations entirely — `lpinfo` from the mediator container fails with "Bad Request" even via Unix socket because the operation no longer exists in the server. `lpinfo` inside the cups container works because it runs backends directly as root. Fix: cups entrypoint writes `devices.txt` and `ppds.txt` to the shared `/run/cups/` volume at startup and refreshes every 60 s; `cups-client.ts` reads those files for `-v`/`-m` queries. IPP fallback kept for remote CUPS ≤ 2.3 servers. Raised CUPS log level to `info`.
   - **Deferred:** Device list is at most 60 s stale. Remote CUPS 2.4 servers cannot do device/PPD discovery (CUPS 2.4 limitation).

## 821. ✅ Fix lpinfo "Bad Request": switch Docker CUPS from TCP to Unix socket
   - **Why:** CUPS 2.4 (Debian bookworm) removed `CUPS-Get-Devices` and `CUPS-Get-PPDs` over TCP for security reasons — `lpinfo -h cups:631 -v` returns HTTP 400 "Bad Request". Fixed by removing `CUPS_HOST: cups:631` and adding `CUPS_SERVER: /run/cups/cups.sock` to mediator env in both compose files, so cups binaries use the already-mounted Unix socket. Also removed `:ro` from the socket volume mount (lpadmin writes to the socket) and added `chmod 777 /run/cups` to entrypoint.sh so www-data can connect.
   - **Deferred:** Remote CUPS server (`printer.server` set in admin UI) still uses TCP; if that server also runs CUPS 2.4 it will hit the same restriction for lpinfo (not fixable from our side — remote lpinfo discovery is then unavailable, print jobs still work via `lp -h`).

## 814. ✅ Fix cups container healthcheck: `lpstat -H` → `lpstat -r | grep 'running'`
   - **Why:** `lpstat -H` on Debian outputs "localhost" (no colon), so `grep -q ':'` always returned 1, marking the cups service permanently unhealthy after 10 retries and blocking mediator startup. `lpstat -r` checks whether the scheduler is actually running, which is the real gate.
   - **Deferred:** Nothing.
